Find Me Online — live embed tracer bullet

Three real ways to surface @alperortac (X) and @AlperTheOrtac (YouTube) content, side by side, with real data. Nothing here is mocked.

Honesty bar: YouTube video IDs/titles/thumbnails below are real, resolved from the channel's public RSS feed (youtube.com/feeds/videos.xml?channel_id=UCNo1thPweKiBs9-gcpvrJhQ) — 5 live videos found, all Shorts. Titles are re-fetched live client-side via YouTube's CORS-enabled oEmbed endpoint (green "live" pill); if that fetch fails, the card falls back to the RSS-resolved title baked in at build time (amber "cached" pill) — never invented copy. X has no public unauthenticated read API — curl-ing twitter.com/x.com returns only a JS app shell with zero server-rendered tweet data, so no real tweet IDs were obtainable without login or a paid API. Every X surface below is therefore either the real twitter-timeline profile widget (loads live, always current) or a real deep-link to the live profile — never a fabricated tweet.

Case 1 — Native feed widgets

fully auto · zero upkeep

Each platform's own embed mechanism, unmodified. X: official twitter-timeline widget (widgets.js). YouTube: a plain grid built from the live RSS feed + live oEmbed titles, platform-styled (not our brutalist chrome) — closest to "just point at the real thing and never touch it again".

Case 2 — Live embeds inside the approved brutalist carousel

real height variance

Same rail/spring/auto-advance as the shipped C1 hero (wireHeroRail, 3.5s auto-advance, pause on hover/drag/focus, easeOutBack spring). YouTube cards are a lite thumbnail facade that swaps to a real youtube.com/embed/<id> iframe on click. The X card is the real twitter-timeline widget dropped in as one wide rail item — there are no real single-tweet IDs to embed with twttr.widgets.createTweet() (see honesty note above), so this is the closest live X representation the rail can host. Rail is deliberately top-aligned, not height-clamped so the real mismatch shows.

Case 2 — proves the risk directly: the X timeline widget is ~600px tall while YouTube facade cards are ~200px, so the rail either clips the widget (ugly) or the row grows to the tallest card (huge dead space next to the short cards). Clicking a YouTube facade swaps in a real iframe with no layout shift since the facade already reserves the 16:9 box — that part works well.

Case 3 — Lite-facade hybrid

recommended to try

Our brutalist chrome (2px ink border, 6px hard shadow, square corners) wrapping real, live content: real YouTube thumbnail + title (RSS + live oEmbed) that swaps to a real embed on click; an on-brand X card that deep-links to the live profile (no fabricated tweet). Uniform card height because we control the layout instead of inheriting the platform's own box.

Case 3 — keeps the carousel look + real content + fast initial load (thumbnails only, no widgets.js/iframe until clicked). Tradeoff: the X card can only ever be a deep-link teaser here, never a real tweet preview, unless the user supplies specific tweet URLs/IDs (X has no free unauthenticated read API to resolve "latest tweet" ourselves) — or unless we pay for the X API to pull real tweet objects server-side.
