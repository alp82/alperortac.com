Likely cause of current lag (from reading useShortsRail.ts): the spring writes rail.scrollLeft = ... on every animation frame, and reads rail.scrollLeft / offsetLeft synchronously elsewhere (currentIndex, atMaxScroll). Assigning scrollLeft repeatedly forces the browser's scroll/layout machinery every frame instead of compositing a transform — that's main-thread work per frame, not GPU-only. It also fights native scroll-snap-type: x proximity on .shorts-rail — the browser's own snap logic can tug against the rAF-driven spring, which is a likely source of the "hard jump" feel on interval advance (see BEFORE rail above, which reproduces this exact pattern).

All motion below animates transform: translate3d() only — no left/margin/scrollLeft writes in the after-rail.
